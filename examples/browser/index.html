Browser Harness

This page verifies BnbZkIdClient loading, config resolution, and the main prove flow in a browser environment: Fixture uses fixtures for both sides; BrevisFixture+PrimusSDK uses static Gateway JSON aligned with GET /v1/config and POST/GET /v1/proof-requests while Primus still talks to PADO; Brevis+PrimusSDK uses the real Gateway HTTP 8038 endpoint together with the Primus SDK.

Naming convention: on the application side, prove.provingParams.businessParams aligns with the Gateway values in GET /v1/config properties[].businessParams and POST /v1/proof-requests businessParams; the full provingParams object is passed into Primus additionParams and may also contain other reserved keys. In Brevis+PrimusSDK, the options come from the live GET /v1/config of the current Gateway base. Prefer npm run dev:browser-harness so requests go through the same-origin /brevis-gateway/ proxy and avoid CORS; if you connect to a remote Gateway directly, the server must enable CORS. In BrevisFixture, the options still match static fixtures/config.json. PADO can go through the /pado-api/ proxy. A Primus browser extension environment is still required.
